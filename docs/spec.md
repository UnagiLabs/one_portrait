# spec.md: ONE Portrait - The Faces of Fans

> **ドキュメント目的:**
> 本ドキュメントは「Sui x ONE Samurai」ハッカソンにおけるプロジェクト「ONE Portrait」の概要仕様書である。AI駆動開発において、システム全体の目的、機能要件、ユーザーフローをAIが理解するためのプライマリ・コンテキストとして機能する。

## 1. プロジェクト概要
「ONE Portrait」は、格闘技団体ONE Championshipの選手とファンを繋ぐ、SuiブロックチェーンとWalrus（分散ストレージ）を活用した非営利の共同制作プラットフォームである。
ファンがアップロードした2,000枚の日常の「写真」が集まった瞬間に、最適化された巨大な選手の「モザイク肖像画」がプログラムによって生成され、1枚の完成アートとしてSui上にNFT化される。

* **コンセプト:** あなたの笑顔が、ヒーローの力になる。
* **ミッション:** 営利を目的とせず、ファンの熱狂と応援の証を最高品質のアートと改ざん不可能なオンチェーンデータとして永続化する。

## 2. ターゲットユーザー

| ユーザー層 | 主な体験 |
| :--- | :--- |
| **ファン** | 写真1枚のアップロードで巨大アート制作に参加し、自分の貢献マスを証明する譲渡不可の**欠片 (かけら) NFT** を受け取る。 |
| **アスリート** | 2,000人のファンの写真で構成された高解像度の肖像画（**Master Portrait NFT**）を受け取る。 |
| **運営 (ONE)** | ファンの熱量を最高品質のデジタルアート作品として生成・蓄積できる。 |

## 3. コア機能要件

### 3.1 摩擦ゼロの参加体験 (zkLogin)
* Web3ウォレット不要。Google/SNSアカウントで即座に参加可能。

### 3.2 選手選択とユニット制
* ファンは参加時に「どの選手のPortraitに参加するか」を選択する。
* 各選手には、on-chain の `Registry` が指す **1つの進行中ユニット**（2,000マス）が存在する。選手名・サムネイルなどの表示メタデータは off-chain catalog で解決する。
* 2,000枚が埋まった時点でそのユニットは確定・生成フェーズに移行する。
* 確定後に同じ選手へ参加したい場合は、運営が次のユニットを手動で開始する。

### 3.3 リビール型モザイク生成システム
「完成の瞬間まで全体像を見せない」ことを体験の核とし、ファンの熱量を一点に集約させる。
1. **待機フェーズ (収集中):** アップロードされた写真は即座にWalrusへ保存されるが、モザイク全体は公開しない。画面上は**残り枚数カウンター**（例: 347/2,000）のみで進捗を示す。投稿者自身の写真は「参加証」として確認できる。
2. **確定・生成フェーズ (2,000枚到達時):** バックエンドが最適な配置アルゴリズムを実行し、1枚の高解像度モザイク画像を生成、Walrusへ保存する。
3. **リビール (公開):** 生成完了と同時に、選手のPortraitが全参加者へ一斉公開される。2,000人のファンが初めて完成形を目にする瞬間を演出として設計する。

### 3.4 オンチェーン・メモリー (Table Data Structure)
* 完成したモザイク画像は、Sui上で1つの **Master Portrait NFT** としてミントされる。MVPでは運営管理アドレスが保有し、将来は選手移管を検討する。
* NFTの内部には、Suiの`Table`（キーバリューマップ）構造を使用し、`blob_id -> placement` の形で各マスの「元の写真のWalrus ID」「投稿者アドレス」「submission_no」「対応する欠片NFT参照」を記録する。

### 3.5 欠片NFT（参加者への配布）
完成した作品をファン自身の誇りとして残すため、2,000人の参加者それぞれに **欠片NFT**（モザイクの1片） を自動発行する。
* **内容:** 自分の投稿写真の `walrus_blob_id`、`unit_id`、`athlete_id`、投稿順を表す `submission_no` (1〜2,000)、投稿時刻、紐づく Master Portrait 参照を保持する。座標 `(x, y)` は直接持たず、完成後に `MasterPortrait.placements` を `blob_id` で逆引きして解決する。
* **Soulbound（譲渡不可）:** 転売・移転を禁止する実装として、Move の型レベルで `store` 能力を付与しない設計を採用する。非営利ミッションと整合し、「ファンの証」としての意味を保全する。
* **発行タイミング:** `submit_photo` と同一トランザクションで即時発行する。投稿が成功した時点で、参加者は自分のKakeraを受け取る。
* **表示:** 欠片NFTを開くと、完成後は自分のマスがハイライトされた状態で Master Portrait 全体を閲覧できる。未完成ユニットでは「完成待ち」として表示する。

### 3.6 マイギャラリー（参加履歴ページ）
zkLoginで認証済みのユーザーが、過去に参加したすべてのPortraitを一覧・再訪できる個人ページを提供する。
* **データソース:** ログインアドレスが保有する **欠片NFT** をSui上で全件クエリし、各NFTから紐づく Master Portrait / 選手 / ユニット番号を解決する。オフチェーンのアカウントDBに依存せず、SBTそのものを参加履歴の正本とする。
* **一覧ビュー:** 参加済みイベント（選手 × ユニット）をカード形式で時系列表示する。`master_id` が未設定のユニットは「完成待ち」、設定済みのユニットはMaster Portraitのサムネイルと自分の担当マスの位置インジケーターを表示する。
* **詳細ビュー:** カードを開くと、Master Portrait全体の中で自分の欠片がハイライト表示され、ズームすると元の投稿写真、マス座標 `(x, y)`、`submission_no`、投稿日時、同ユニットの参加者数などのメタデータを確認できる。元の投稿写真が取得できない場合でも、完成作品と欠片メタデータは継続表示する。
* **共有:** 詳細ビューから「自分の欠片が刻まれた作品」として、SNS共有用のOG画像（ハイライト付きプレビュー）を生成できる。

### 3.7 運用・公開ポリシー
* **原画像の公開性:** `walrus_blob_id` は on-chain 状態に記録されるため、UI上で隠していても private にはならない。`blob_id` を知る者は原画像を取得できる前提で設計する。
* **同意:** 投稿前に「原画像は `blob_id` を知る者が取得できる」旨を明示し、参加者の同意を必須とする。
* **MVP運用:** 不適切画像のモデレーションフロー、投稿後の差し替え・削除、長期未達ユニットの timeout / cancel は MVP では扱わない。
* **保存保証:** プロダクト意図としては参加履歴の再訪性を重視するが、ハッカソンMVPではファン投稿原画像の長期保持を保証しない。完成モザイクとオンチェーンの参加記録を優先して残す。

## 4. ユーザーフロー
1. **アクセス & ログイン:** QRコードからアクセスし、zkLoginで認証。
2. **選手選択:** 参加したい選手のPortraitユニットを選ぶ。
3. **同意 & 投稿:** 原画像公開性に同意したうえで、スマホで写真を撮影しアップロード。画像は即座にWalrusへ保存される。
4. **Kakera即時受領:** `submit_photo` の成功と同時に、自分専用の **欠片NFT**（Soulbound、`submission_no` 入り）が自分のウォレットに届く。
5. **待機体験:** 全体モザイクは非公開のまま、残り枚数カウンターで完成までの進捗を見守る。自分の投稿写真は「参加証」として確認できる。
6. **リビール演出:** 2,000枚に達した瞬間、バックエンドがモザイクを生成し、全参加者の画面で完成した肖像画が初めて一斉に公開される。
7. **Master確定:** 完成モザイクが Master Portrait NFT として記録され、自分のKakeraは `blob_id` 逆引きによって担当マスと紐づく。
8. **再訪 (マイギャラリー):** 後日サイトへアクセスすると、保有する欠片NFTから過去の参加履歴が自動復元される。元写真が取得できない場合でも、完成作品と欠片情報は確認できる。

## 5. Sui/Walrusテクノロジーの活用意義
* **Walrus (Storage):** 完成モザイクと目標画像は長期保存し、ファン投稿写真は `blob_id` を正本識別子として保持する。ハッカソンMVPでは各原画像の長期可用性までは保証しない。
* **Table (Data Structure):** 2,000件の履歴データを Master Portrait NFT 内にベタ書き（vector）するのではなく、Suiの`Table`でキーバリューとして保持することで、ガス代を最小化しつつスマートなオンチェーン記録を実現する。
* **Soulbound NFT (Move能力設計):** 欠片NFTは Move の能力（`key` のみ付与し `store` を外す、または custom transfer policy を用いる）によって譲渡不可を型レベルで保証する。Suiのオブジェクト中心モデルだからこそ可能な、安全で非営利な「ファンの証」を実現する。
